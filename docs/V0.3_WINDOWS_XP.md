# v0.3 — Windows XP install USB

## Why XP is its own mode

XP install media predates USB-aware Windows installers by a decade. The boot chain is fundamentally different from Vista+:

| Stage                  | Vista / Win 7+ (`--type=windows`)  | Windows XP (`--type=windows-xp`)               |
|------------------------|------------------------------------|------------------------------------------------|
| MBR boot code          | `ms-sys --mbr7`                   | `ms-sys --mbr` (Win 2000/XP/2003 MBR)          |
| FAT32 PBR              | `ms-sys --fat32pe` (loads `bootmgr`) | `ms-sys --fat32nt` (loads `NTLDR`)             |
| Stage-2 loader         | `bootmgr` → loads `BCD` from BCD store | `NTLDR` → reads `boot.ini` → invokes setupldr  |
| Source media expected  | USB or DVD (designed for both)     | CD / floppy (USB requires modifications)       |
| Install file path      | `sources/install.wim`              | `i386/` directory tree                         |

After NTLDR runs, XP's text-mode setup phase tries to load drivers and copy install files. It expects the source media to behave like a CD-ROM. A USB stick *is* a mass-storage device but XP's stock `txtsetup.sif` lists USB drivers only as input devices — they're loaded, but not recognized as boot-bus extenders. The install reaches the "Please wait while Setup loads files..." stage, then the second-stage reboot fails to find the install source.

## The WinSetupFromUSB recipe

This is the canonical solution circa 2007-2009 (Ilko-y, boot-land.net). The modifications are:

### Required: `txtsetup.sif` modification

Move five driver lines from `[InputDevicesSupport.Load]` to `[BootBusExtenders.Load]`:

```
usbehci, usbehci, usbehci.sys
usbohci, usbohci, usbohci.sys
usbuhci, usbuhci, usbuhci.sys
usbhub,  usbhub,  usbhub.sys
usbstor, usbstor, usbstor.sys
```

Also add corresponding entries in `[BootBusExtenders]`:

```
usbehci  = "USB 2.0 Enhanced Host Controller"
usbohci  = "USB Open Host Controller"
usbuhci  = "USB Universal Host Controller"
usbhub   = "USB Standard Hub"
usbstor  = "USB Mass Storage"
```

Without this, text-mode setup loses the USB source after the reboot.

### Recommended: `WaitBT.sys` and `Wait4UFD.sys` drivers

Third-party waiters that delay the kernel until the BIOS finishes initializing USB on the target machine. Without them, some hardware throws `0x7B INACCESSIBLE_BOOT_DEVICE` on the post-text-mode reboot.

Provenance: originally written by ilko-y for boot-land.net (2009). Free for redistribution but not GPL or MIT-licensed. usbwin's approach for v0.3: **user-provided via `--xp-waiters /path/to/drivers/`** so we don't bundle and don't have to argue about license status.

### Optional: `winnt.sif` (answer file)

If present at the partition root, XP setup reads it to skip prompts (EULA, partitioning, product key, computer name, etc.). usbwin generates a minimal `winnt.sif` if the user supplies `--xp-unattended`.

## Destructive-wipe entry (`\WIPE.DAT`)

XP setup will happily install onto a target HDD that still has GPT
headers, foreign partition tables, or other leftovers from a previous
OS — but the resulting install boots into a "corrupted-looking" state
(protective-MBR confusion, orphan GPT primary header in the slack of
what XP thinks is its MBR, etc.). The fix is to wipe the first 1 MiB
of the target before letting XP setup repartition.

We bake the wipe into the existing NTLDR `boot.ini` menu as a third
entry — no grub4dos, no SYSLINUX, no second bootloader. The entry
chainloads a 512-byte mkmsbr-supplied bootsector (`\WIPE.DAT`,
`bootrec::WIPE_BOOTSECT_BOOT`, source in
`mkmsbr/boot-asm/wipe_bootsect.asm`).

Final `boot.ini` (CRLF; default = entry 1):

```ini
[boot loader]
timeout=10
default=C:\$WIN_NT$.~BT\BOOTSECT.DAT

[operating systems]
C:\$WIN_NT$.~BT\BOOTSECT.DAT="1st, text mode setup"
multi(0)disk(0)rdisk(1)partition(1)\WINDOWS="2nd, GUI mode setup"
C:\WIPE.DAT="3rd, wipe internal HDD (destructive)"
```

### Safety model

The wipe bootsector receives BIOS-supplied `DL` (= the USB drive — the
disk we MUST NOT touch). It computes `target = DL XOR 1`, flipping the
low bit to map 0x80 ↔ 0x81. This handles both BIOS conventions
(USB-at-0x80 + HDD-at-0x81, or the inverted layout some legacy BIOSes
use) without ever wiping the USB.

Before any destructive operation:

1. `INT 13h fn 0x41` (check extensions) on the target — abort with
   `no target` if the drive doesn't exist or doesn't support extended
   I/O.
2. `INT 13h fn 0x48` (get extended drive params) reads the target's
   total sector count. Convert to MiB via 32-bit DIV (16-bit DIV
   overflows on any disk larger than ~64 GB; on real hardware that
   raises `#DE` and hangs the CPU at the INT 0 vector with a flashing
   cursor — see the 2026-05-20 first-hardware-test failure mode).
3. Display:

   ```
   USBWIN WIPE
   target=0xNN size=NNNNNN MiB
   USB=0xNN safe
   Y=wipe?
   ```

   User sanity-checks the size matches the expected target HDD before
   committing.
4. Single `Y`/`y` keypress confirms. Any other key cancels.
5. On confirm: 2048 `INT 13h fn 0x43` (extended write) calls zero
   LBA 0..2047 of the target (1 MiB). Then `INT 19h` warm-reboot.

### v1 limitations

- **Single internal HDD assumed.** Multi-HDD machines (USB at 0x80,
  internal disks at 0x81 + 0x82 + …) wipe only `DL XOR 1`. A v2 would
  enumerate 0x80..0x87 and present a picker; this needs a multi-sector
  bootsector since a picker UI doesn't fit in 512 bytes alongside the
  size readout + division code.
- **GPT backup at last LBA not cleared.** The wipe covers MBR +
  protective-MBR slack + the start of any GPT primary header. The
  GPT backup header at the disk's last LBA stays. XP doesn't speak
  GPT so it doesn't matter for the install path; downstream tools
  may still see an orphan GPT backup.
- **`boot.ini` default is text-mode**, not wipe — accidental Enter
  at the NTLDR menu does NOT wipe. The wipe path requires both
  arrow-down-to-select-entry-3 AND a Y at the wipe prompt.

### Implementation

- `mkmsbr/boot-asm/wipe_bootsect.asm` — 512-byte NASM source.
- `mkmsbr::WIPE_BOOTSECT_BOOT` — `include_bytes!`'d blob, exposed
  the same way as `MBR_XP_BOOT` / `FAT32_PBR_NTLDR_MULTI_BOOT`.
- `pipeline::xp_staging::stage_wipe_bootsect()` — writes the blob to
  `\WIPE.DAT` at the partition root, asserts size = 512 and magic
  `0x55AA` before writing.
- `pipeline::xp_staging::BOOT_INI` — three-entry `boot.ini` const.
- `pipeline::windows_xp::run()` — calls `stage_wipe_bootsect` after
  `stage_root_boot_files`.

## NT 3.x / NT 4.0 — out of scope

Pre-2000 NT family doesn't support USB at all in the installer phase. NT 3.x has zero USB support; NT 4.0 had third-party USB drivers but never first-class. Installing NT 3.x / 4.0 from USB is essentially impossible without VM-then-image-the-disk shenanigans. Out of scope for usbwin.
