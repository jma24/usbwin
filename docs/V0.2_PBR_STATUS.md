# v0.2 Windows-mode PBR: status and path to v1.0

## What v0.2 does

For Windows 7+ install USBs, usbwin handles the full pipeline natively *except* the boot-record bytes themselves:

| Step                          | Implementation in v0.2          |
|-------------------------------|--------------------------------|
| Partition table               | usbwin (Rust, `build_mbr`)     |
| FAT32 format                  | `newfs_msdos` (macOS native)   |
| ISO mount + file copy         | usbwin (Rust, chunked)          |
| MBR boot code                 | **`ms-sys --mbr7`** (external)  |
| FAT32 PE PBR (BOOTMGR-loading)| **`ms-sys --fat32pe`** (external) |
| Unmount + eject               | usbwin (`diskutil`)             |

The two `ms-sys` invocations are the pragmatic concession in v0.2. They write the exact byte sequences your Dell E6410 already accepted in [`FIELD_FINDINGS_2026_05_18.md`](FIELD_FINDINGS_2026_05_18.md) §8 — the validated end-to-end recipe.

## Why ms-sys, not a clean-room PBR

A long debug session attempted a clean-room single-sector FAT32 PBR ([`boot-asm/fat32_pbr.asm`](../boot-asm/fat32_pbr.asm)). It fails on real legacy-BIOS hardware (Dell E6410) AND in QEMU SeaBIOS with `INT 13h fn 0x42` returning `AH=0x01` (= LBA out of bounds, per SeaBIOS source `src/disk.c`). Root cause findings:

1. **Multi-sector PBR architecture.** Microsoft's production FAT32 PE PBR is 16 sectors, not 1. The first sector loads, then it uses INT 13h ext to read the remaining 15 sectors into 0x7E00..0x9BFF. Those 15 sectors contain the rest of the BOOTMGR-loading logic. Source: [thestarman's Win 7 VBR analysis](https://thestarman.narod.ru/asm/mbr/W7VBR.htm). FIELD_FINDINGS §3 had already flagged this; we confirmed empirically.

2. **INT 13h ext check + CHS fallback.** Microsoft's PBR calls `fn 0x41` (check extensions) first. If extensions are absent — or, in our observed case, if they're present but `fn 0x42` returns `AH=0x01` for the LBA we want — falls back to `fn 0x02` (CHS reads). The CHS path uses BIOS-translated geometry that handles the "USB-treated-as-floppy" case ([RMPrepUSB §27](https://rmprepusb.com/tutorials/027-diagnose-how-your-bios-boots-usb-drives/) covers this BIOS-quirk class).

3. **BIOS-as-floppy emulation.** Many pre-2012 BIOSes (and possibly the Dell E6410) report a USB stick's geometry as ~1.44 MB regardless of physical capacity. Our PBR's `fn 0x42` reads at LBA 32602 are then "out of bounds" of the emulated floppy size. The MBR's read at LBA 2048 squeaks under the threshold by coincidence.

The clean-room PBR could be rewritten to handle this (16-sector layout + CHS fallback + BIOS geometry query), but that's substantial NASM work and needs reliable test hardware to iterate. Deferred to v1.0.

## Installation requirements for v0.2 Windows mode

Beyond the v0.1 base (Rust + NASM):

```sh
# Build ms-sys from source (one-time)
git clone https://gitlab.com/cmaiolino/ms-sys.git /tmp/ms-sys
cd /tmp/ms-sys && make
sudo cp bin/ms-sys /usr/local/bin/
```

Or, without `sudo`:
```sh
export USBWIN_MS_SYS=/tmp/ms-sys/bin/ms-sys
```

usbwin's `pipeline::diskutil::find_ms_sys` checks `USBWIN_MS_SYS`, then `/usr/local/bin/ms-sys`, then `/opt/homebrew/bin/ms-sys`, then PATH. If none resolve, Windows-mode bails up front with install instructions.

## Path to v1.0 (clean-room PBR, no ms-sys dependency)

Three options, in increasing fidelity-to-Microsoft order:

1. **Single-sector + CHS fallback (BootProg-style).** Study [Alexey Frunze's BootProg](https://github.com/alexfru/BootProg) (BSD-2 clean-room, MIT-compatible) for the algorithm shape. Reimplement clean-room. ~weekend of work. Practical limit: only works for partitions whose CHS-addressable area covers BOOTMGR's location (almost always true for Win 7 install USBs; install.wim past 8 GB would not be reachable via CHS but is reached after bootmgr loads its own ext-disk driver).

2. **Multi-sector clean-room PBR.** Layout matching Microsoft's 16-sector scheme: sector 0 with BPB + ext-check + CHS-fallback + initial loader; sectors 1-15 with extended FAT walking and bootmgr loading. ~weeks of work. Byte-equality with ms-sys output checkable via `tests/compare_mssys.rs` (already scaffolded in `usbwin-boot/Cargo.toml` `compare-mssys` feature).

3. **Build ms-sys at install time.** usbwin's `cargo install` triggers a Make-based build of vendored ms-sys source. Keeps the dependency but bundles it. Legal grey for GPL→MIT bundling; probably wouldn't make it past public-release review.

Recommendation when revisiting: **option 1**. BootProg-style is the smallest viable clean-room implementation, well-documented, and proven on real hardware. The multi-sector approach is overkill for the install USB use case.

## What v0.2 ships

`cargo build --release --features usbwin-boot/embed-boot-asm` produces a working `target/release/usbwin` binary. Hybrid mode is fully tested. Windows mode requires `ms-sys` installed (per above) and is byte-validated against the FIELD_FINDINGS §8 recipe.
